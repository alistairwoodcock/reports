# FurBaby Social Backend - CI/CD & DevOps Assessment

**Date:** 2026-04-18

---

## Pipeline Overview

| Stage | Runner | Trigger | Target |
|-------|--------|---------|--------|
| Secret scan (ggshield) | Drone CI | push/PR on dev/stage/master | GitGuardian API |
| Tests | Drone CI | push/PR on dev/stage | pytest |
| Build Dev | Drone CI + ECR plugin | push on `dev` | ECR → `fastapi-dev` |
| Deploy Dev | Drone CI + `kudato/drone-ecs-deploy` | push on `dev` | ECS svc `fur-baby-fastapi-dev` (us-east-1) |
| Build Stage | Drone CI + ECR plugin | push on `stage` | ECR → `fastapi-stage` |
| Deploy Stage | Drone CI + ECS plugin | push on `stage` | ECS svc `fur-baby-fastapi-stage` |
| Build Prod | Drone CI + ECR plugin | push on `master` | ECR (prod account) → `fastapi-prod` |
| Deploy Prod | Drone CI + ECS plugin | push on `master` | ECS cluster `prod-ecs` (ap-southeast-2) |
| Slack notify | Drone CI | success/failure on dev/stage/master | Appello Slack |

Runtime image: `public.ecr.aws/n6m8k5u6/appello-baseimage:3.12.7-slim-bookworm-uv`.

## Strengths

1. **Consistent single-vendor CI** — Drone across environments (unlike the mobile repo which splits Drone/Codemagic).
2. **Per-environment AWS accounts** — prod builds land in a separate account (649153120602) and region (ap-southeast-2) from dev/stage (896549618545, us-east-1). Good blast-radius isolation.
3. **Runtime secrets via AWS Secrets Manager** — `retrieve-secrets.py` is invoked from `run.sh` at container start, so secrets are never baked into the image.
4. **Firebase credentials bootstrapped at start** — `firebase_keys.py $ENVIRONMENT` fetches the appropriate credential per environment.
5. **Secret scanning in CI** — `ggshield` runs on every push/PR as the first step (`.drone.yml:69–78`).
6. **Migrations run on deploy** — `run.sh` calls `alembic upgrade head` before starting gunicorn.
7. **Elastic APM + Logstash** — integrated in `main.py` (APM) and via the `appello-logger` / `python-logstash-async` packages.
8. **Slack notifications** — build success/failure pings wired for each of dev/stage/master.

## Gaps & Issues

### CRITICAL: No Static Analysis in CI

`ruff` and `pyright` are configured in `pyproject.toml` and ruff is wired into pre-commit — but **neither runs in CI**. Pre-commit is easily bypassed with `--no-verify`. There is no CI-level gate preventing lint violations or type regressions from landing.

```yaml
# missing from .drone.yml:
- uv run ruff check .
- uv run pyright .
```

---

### CRITICAL: No Test Coverage on the Prod Build

The `tests` step is conditional on `dev` and `stage` branches only (`.drone.yml:81–100`). A direct merge to `master` (prod) builds and deploys **without re-running the test suite**. Combined with the lack of a manual approval gate (see below), this means the prod pipeline has no automated quality gate.

---

### HIGH: No Deploy Approval Gate for Prod

Pushes to `master` automatically:
1. Build a Docker image,
2. Push to the prod ECR account,
3. Call ECS update-service,
4. Roll traffic to the new task.

There is no manual confirmation step and no promotion mechanic (e.g. "stage image → prod") — every prod deploy is a fresh build from source. A typo on `master` rolls to production in ap-southeast-2.

**Recommendation:** Either gate the prod deploy behind a `promote` event / tag, or use a Drone `when: event: [promote]` step. Alternatively, promote the **same image digest** from stage to prod rather than rebuilding.

---

### HIGH: No Test Coverage Reporting or Threshold

The test invocation is a plain `python -m pytest tests/` — no `--cov`, no `--cov-fail-under`, no artefact. With ~13 active tests against ~6 000 LOC of application code, coverage is certainly low but is not measured. See the Testing Strategy report for a concrete plan.

---

### HIGH: No Dependency Vulnerability Scanning

Neither `pip-audit`, `safety`, Snyk, nor Dependabot is configured. The repo has a `requirements.txt` generated by `uv export`, so automating this is straightforward. ggshield covers leaked secrets but not vulnerable dependencies.

---

### MEDIUM: Single Non-Rollback Deploy

The ECS deploy is a rolling update via `kudato/drone-ecs-deploy`. There is no:
- Blue/green or canary setup
- Automated rollback on health-check failure
- `/healthcheck` depth — current endpoint returns 200 statically (see scalability report §7), so ECS believes a task with broken Redis/Firebase is healthy.

**Recommendation:** Add a deep health check (DB + Redis ping) used by the ECS target-group health check. Consider `appspec.yml` / CodeDeploy blue-green for prod, or at minimum a runbook for rollback.

---

### MEDIUM: Migrations Run in-Container with No Safety Net

`run.sh:10` runs `alembic upgrade head` *inside* every task container at startup:
- If N tasks start simultaneously, N parallel migration attempts race each other (Postgres DDL locking saves you most of the time, but this is not an intentional design).
- Failed migrations leave tasks crash-looping — ECS will keep restarting rather than alerting.
- There is no `alembic downgrade` rollback story for a bad migration.

**Recommendation:** Run migrations as a dedicated one-shot ECS task before updating the service, not as part of task startup. Add an explicit "migration ran cleanly" signal before the deploy proceeds.

---

### MEDIUM: Version Management

`pyproject.toml:3` is pinned at `version = "0.1.0"` and never incremented. Docker tags are static per environment (`fastapi-dev`, `fastapi-stage`, `fastapi-prod`) — there is no SHA-tagged image, which makes rollback ("redeploy yesterday's image") impossible without rebuilding.

**Recommendation:** Tag images with `{env}-{git_sha}` in addition to the moving env tag; keep the last N images retained in ECR; bump `pyproject.toml` version from CI or drop it.

---

### MEDIUM: Pre-commit Gaps

`.pre-commit-config.yaml` runs:
- `ruff --fix-only`
- `ruff-format`
- `uv export` (requirements.txt regen)

It does **not** run:
- `pyright`
- `pytest`
- `commitlint`
- `pip-audit` / `safety`

Since these also don't run in CI, nothing enforces them anywhere.

---

### LOW: Dev/Stage Drift on AWS Accounts

Dev and stage share the same AWS account and ECS cluster (`appello-cluster-dev-stage-1`). A runaway stage deploy can affect dev. Acceptable for cost reasons; worth noting.

---

## Recommendations

### Quick Wins
1. Add `uv run ruff check .` and `uv run pyright .` to `.drone.yml`.
2. Add `--cov --cov-report=xml --cov-fail-under=60` to the pytest step; publish coverage as a Drone artefact.
3. Run tests on the prod build leg too (cheap insurance).
4. Add `pip-audit` (or Dependabot) as a recurring job.
5. Tag built images with the git SHA alongside the env tag.

### Next Steps
6. Gate prod deploys behind a manual promotion event.
7. Move Alembic migrations to a dedicated pre-deploy task.
8. Deepen the `/healthcheck` endpoint and wire it into the ECS target-group health check.
9. Add a rollback runbook and test it.

### Longer Horizon
10. Explore blue-green / canary for prod.
11. Automate dev/stage DB schema diff checks before prod merge.
12. Add synthetic smoke tests post-deploy (hit `/healthcheck` and one authenticated endpoint).
13. Add performance regression testing (p50/p95 on representative endpoints).
