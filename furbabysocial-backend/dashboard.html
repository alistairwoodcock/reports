External Dependencies & Caching

External Services

The backend depends on eight external services. Some have to be fast for the app to feel fast; others could be decoupled so users never wait on them.

Service What it's used for What users notice if it's slow Avoidable?
PostgreSQL Main database Everything stalls Inherent
Redis Caches user lookups after login Small slowdown; app still works Already good
Firebase Login tokens + push notifications Login and posting feel sluggish Partly — push notifications don't need to block
Twilio Sending OTP codes by SMS Sign-in stalls waiting for SMS delivery Yes — should be fire-and-forget
AWS S3 Storing user-uploaded photos Photo uploads stall; feed still loads Yes — phone could upload directly with a short-lived link
Centrifugo Real-time updates to the feed Creating a post stalls on broadcast Yes — broadcast should be fire-and-forget
SMTP (email) Admin emails only Admin workflows delayed; users unaffected Yes, but low priority
Elastic APM Performance monitoring No effect on users Already background

Take-away: the biggest wins are Twilio (sign-ins shouldn't wait for SMS delivery), S3 (photo uploads shouldn't travel through the backend), and Centrifugo (broadcasting a new post shouldn't block the caller). Fixing these makes sign-in, posting, and photo-upload feel instant even when those services are slow.
